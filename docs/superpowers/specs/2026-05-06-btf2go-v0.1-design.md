# btf2go v0.1.0 — Design Spec

**Date:** 2026-05-06
**Status:** approved for implementation planning
**Module:** `github.com/danigoland/btf2go`
**License:** Apache-2.0

## 1. Positioning

btf2go takes any pre-compiled eBPF ELF (clang, rustc/Aya, zig) and emits Go struct definitions matching the kernel-visible memory layout. It complements `cilium/ebpf`'s `bpf2go` by removing the C-compilation step from the pipeline: feed in any `.elf`, get out Go types.

btf2go is **not** a `bpf2go` replacement. Use `bpf2go` for the "I write C and want a full Go skeleton" path. Use btf2go for the "I built my eBPF program in Rust/Zig/CO-RE and just need correct Go types" path.

### What `bpf2go` already does (architecture-doc correction)

The original architecture doc framed the gap as "bpf2go parses C headers." That is no longer accurate: `cilium/ebpf` PR #552 (Jan 2022) added BTF-driven type generation, and the current bpf2go README states *"bpf2go generates Go types for all map keys and values by default."* The actual gap btf2go fills is that bpf2go **wraps** the BTF-extraction step inside a `clang`-orchestration pipeline — it expects to compile a `.c` source file. There is no idiomatic path through bpf2go for "I already have an `.elf` from rustc or zig, just give me Go types."

### Differentiation features (things bpf2go does not currently emit)

- Generated **bitfield accessor methods** (the most concrete win)
- Generated **union accessor methods** for typed views over `[N]byte` storage
- Generated `Pointer[T]` phantom-typed wrapper for kernel-address fields, declared inline in the generated file (no runtime dependency on `pkg/btf2go`)
- Name sanitizer for Rust namespace mangling (`my_mod::Foo` → `MyModFoo`) and Zig equivalents

## 2. Scope (v0.1.0)

In scope:
- Structs, arrays, pointers, enums, **unions with accessor methods**, signed and unsigned integers
- Bitfields rendered as a raw `[N]byte` storage field **plus generated bit-extraction accessor methods**
- Whitelist-based type selection (map key/value types auto-included; `--type` adds more; recursive dependency closure)
- Name sanitization for namespace-mangled BTF names; anonymous structs/unions named `<ParentField>Anon<N>`
- Single self-contained output `.go` file, formatted via `go/format` (or written unformatted to `--out` with a stderr warning if formatting fails)

Out of scope for v0.1.0:
- Loader / `*ebpf.CollectionSpec` generation, embedded ELF, typed map handles, `Load*()` functions. Rationale: `bpf2go` already does this well; the loader path already works for any-language ELFs via `cilium/ebpf` directly. Adding it would triple surface area and put btf2go in direct overlap with bpf2go's strongest axis. Revisit in v0.3+ if demand exists.

Endianness: BTF does not carry an endianness marker for value bytes. The generator targets the host endianness of the build that produced the `.elf`. For the overwhelmingly common little-endian case (linux/amd64, linux/arm64), one generation suffices. Document this; do not engineer around it in v0.1.

## 3. Architecture: 5-Phase Pipeline

```
ELF file
   │
   ▼
[Phase 1] Ingestion          internal/btfparser
   │   btf.LoadSpec(path) → *btf.Spec
   ▼
[Phase 2] Resolution + Sanitization   internal/btfparser
   │   walk Maps → seed type set
   │   add explicit --type names
   │   recursive dep closure
   │   sanitize names (strip ::, anonymous → ParentFieldAnonN)
   ▼
[Phase 3] Type Engine → IR            internal/types
   │   btf.Int → intN/uintN (Encoding.IsSigned)
   │   btf.Array → [N]T
   │   btf.Pointer → Pointer[T] (locally generated wrapper)
   │   btf.Enum → Go type alias + const block
   │   btf.Union → [MaxSize]byte + accessor methods
   │   btf.Struct → IR struct (fields not yet aligned)
   ▼
[Phase 4] Alignment Calculator        internal/align
   │   bit→byte, padding insertion, packed-struct downgrades,
   │   bitfield block detection + accessor planning
   ▼
[Phase 5] Codegen                     internal/generator
   │   text/template → string
   │   go/format.Source → bytes
   │   write to --out (warn + dump unformatted on format failure)
   ▼
generated .go file
```

Two non-obvious decisions:

- `internal/align` imports nothing from `github.com/cilium/ebpf/btf`. It operates purely on the IR plus a small `goAlign(kind) int` helper. This is what makes the alignment logic unit-testable with mock IR.
- The `Pointer[T]` wrapper is emitted into every generated file (one declaration at the top of the file), not imported from a runtime package. Generated files are self-contained.

The library API is `btf.LoadSpec(path)` (it auto-detects ELF). The original arch doc's `LoadSpecFromELF()` was an outdated name.

## 4. CLI Surface

```
btf2go generate \
  --elf path/to/prog.elf       # required
  --pkg events                 # required, target Go package name
  --out generated.go           # required, single output file
  --type EventA --type EventB  # repeatable; map K/V types auto-included
  [--no-map-types]             # opt out of automatic map K/V inclusion
```

Selection algorithm: start with (`--type` set ∪ map K/V types unless `--no-map-types`), then take the recursive dependency closure. Unknown `--type` is a hard error. Output is a single self-contained file.

Cobra is the CLI framework. `generate` is the only command in v0.1.

## 5. IR Shape (the contract between phases)

```go
// internal/types
type GoFile struct {
    Package string
    Enums   []GoEnum
    Structs []GoStruct
    Unions  []GoUnion
}

type GoStruct struct {
    Name      string
    Size      uint32       // total size in bytes (post-alignment)
    Fields    []GoField    // includes synthetic padding fields
    Bitfields []GoBitfieldBlock // accessors to emit alongside the struct
}

type GoField struct {
    Name   string  // "_padN" for synthetic padding, "_bfN" for bitfield storage
    Kind   Kind    // Primitive | Array | Pointer | NamedStruct | NamedUnion | RawBytes
    GoType string  // rendered Go type, e.g. "uint64", "[16]byte", "Pointer[Foo]"
    Offset uint32  // byte offset within parent
    Size   uint32  // byte size
    IsPad  bool
}

type GoBitfieldBlock struct {
    StorageField string         // e.g. "_bf0"
    StorageSize  uint32         // bytes
    Accessors    []GoBitAccessor
}

type GoBitAccessor struct {
    Name      string // method name (sanitized BTF field name)
    BitOffset uint32 // within storage block (0 = LSB of first byte)
    BitWidth  uint32
    Signed    bool
    GoType    string // uint8/uint16/uint32/uint64 or signed equivalents
}

type GoEnum struct {
    Name       string
    Underlying string         // uint32 by default, uint64 if any value > 2^32-1
    Values     []GoEnumValue
}

type GoEnumValue struct {
    Name  string
    Value int64
}

type GoUnion struct {
    Name      string
    Size      uint32
    Storage   string // "_data [N]byte"
    Accessors []GoUnionAccessor
}

type GoUnionAccessor struct {
    Name   string // member name → As<Member>() / SetAs<Member>(v T)
    GoType string
    Size   uint32 // <= union size
}
```

Phase 4 takes a partially-populated `GoFile` from Phase 3 and mutates it: inserts `_padN` fields, downgrades misaligned primitives to `[N]byte`, populates `Bitfields`. Phase 5 only reads the IR.

## 6. Alignment Rules

For each struct, walk members in declaration order:

1. `byteOffset = member.Offset / 8`
2. If any member has `BitfieldSize > 0` OR `Offset % 8 != 0`: collapse the contiguous bitfield run into a single `_bfN [storageBytes]byte` field. Emit one `Get<Field>` and one `Set<Field>` accessor per bitfield, with bit-shift/mask math computed from `Offset` and `BitfieldSize`.
3. Track `goCursor` (simulated byte offset within the struct being built). If `byteOffset > goCursor`: insert `_padN [byteOffset - goCursor]byte`, advance cursor.
4. After placing a primitive field, check Go natural alignment: `if goCursor % goAlign(kind) != 0` (i.e., BTF placed it where Go's `gc` compiler would not), downgrade the primitive's Go type to `[size]byte`. This forces the exact layout.
5. After all fields, if total field bytes < `struct.Size`: append `_padN` to fill.

`goAlign` for linux/amd64 + linux/arm64 (the targets eBPF cares about):

| Kind | Alignment |
|------|-----------|
| u8/i8/bool | 1 |
| u16/i16 | 2 |
| u32/i32 | 4 |
| u64/i64/Pointer[T] | 8 |
| array[N]T | align(T) |
| nested struct/union | max alignment of any member |

`btf.Int` signed-vs-unsigned uses `IntEncoding.IsSigned()`. Char-encoded ints map to `int8`/`uint8` (not `byte`/`rune`) for layout clarity.

## 7. Generated Output Layout

Single file, sections in this order:

1. `// Code generated by btf2go. DO NOT EDIT.` header with source ELF path and tool version
2. `package <pkg>`
3. `import` block. `import "unsafe"` is included whenever the file emits any union (the accessor methods use `unsafe.Pointer` casts on the storage byte array). Otherwise the import block is omitted.
4. `type Pointer[T any] uint64` declaration (always emitted)
5. Enum types and their `const` blocks
6. Union types with `[N]byte` storage and `As<Member>` / `SetAs<Member>` accessors using `unsafe.Pointer` casts
7. Struct types with all fields including synthetic padding, plus `Get<Field>`/`Set<Field>` bitfield accessors as methods

Output is run through `go/format.Source`. On formatting failure, the unformatted bytes are still written to `--out` and a warning is printed to stderr including the formatter error — this lets users debug template issues without losing output.

## 8. Testing Strategy

Three layers:

- **Unit (CI):** `internal/align` table-driven tests on hand-crafted `GoFile` IRs covering: tight packing, leading/trailing padding, a single bitfield, a multi-bitfield run straddling bytes, a packed `uint64` at offset 4, nested struct alignment. No `btf.*` types touched here — the alignment package is pure IR-in / IR-out.
- **Golden output (CI):** committed `.elf` fixtures under `tests/fixtures/{c,rust,zig}/`. Tests run btf2go in-process and compare output to committed `.golden.go`. `UPDATE_GOLDEN=1 go test ./...` regenerates goldens.
- **Layout verification (CI):** committed golden `.go` files are compiled into a verifier program that uses `unsafe.Offsetof`/`Sizeof` on every field and asserts against a JSON sidecar (per fixture) of expected offsets and sizes.

The toolchain (clang, cargo + Aya, zig) is invoked **only by `make fixtures`** (local). CI runs pure Go. This decouples the generator from upstream toolchain churn — broken bpf-linker releases do not redden btf2go PRs.

Initial fixture set (one per language):
- `events_c.elf` — a struct with a nested union, an array, a bitfield, and a pointer
- `events_rust.elf` — equivalent struct compiled via Aya, exercising namespace mangling
- `events_zig.elf` — equivalent struct compiled via zig, exercising packed alignment

Each fixture ships with: `.elf`, `.golden.go`, `.layout.json`.

## 10. Risks and Open Questions

- **Cilium could absorb this in a day.** A `bpf2go --skip-compile --elf foo.o` flag would close most of the gap. Differentiation must come from output quality (union/bitfield accessors, name sanitization) and the no-clang-required ergonomics for non-C workflows. Plan: build the differentiation features in v0.1 so the project stands on its own merits even if upstream extends bpf2go.
- **Upstream toolchain churn.** Rust eBPF with BTF is still bumpy at the LLVM/bpf-linker layer (e.g., LLVM `BTFDebug` issue #143361 from June 2025 about wrapper-struct BTF). Diagnostics need to clearly distinguish "this BTF input is malformed" from "btf2go bug." Out of scope for the v0.1 plan but worth a tracking issue.
- **Anonymous-struct naming collisions.** The `<ParentField>Anon<N>` scheme is unambiguous but ugly when nested. Acceptable for v0.1; revisit if real-world output is unusable.
- **Endianness.** Big-endian eBPF deployment (s390x) is rare; if a user hits it, document "regenerate on a same-endianness host" rather than emitting per-endianness files. Reconsider only if a real user surfaces.

## 11. Out of Scope (Explicit Non-Goals for v0.1)

- Loader scaffolding (`*ebpf.CollectionSpec` generation, embedded ELF, typed map handles, `Load*()` functions)
- CO-RE relocation handling (consume relocated ELFs as input; do not emit relocation logic)
- Cross-endianness output
- Per-architecture output files
- Watch mode, IDE integration, language-server features
- Reading BTF from `/sys/kernel/btf/vmlinux` (kernel-type generation is an `aya-tool`-style tool, not this one)
