### Orchestration gotchas

1. **Daytona CLI auth (401)**: The Daytona CLI v0.173.0 and the Daytona MCP both returned 401, despite the JWT config having a future expiry (2026-05-11). Root cause: the profile API key was null and the JWT token required `X-Daytona-Organization-ID` header which neither the CLI nor the MCP sent. Workaround: direct HTTP calls to `https://app.daytona.io/api/` and toolbox proxy with the org header.

2. **MINIMAX_API_KEY not passed through create**: The sandbox create API call accepted the env dict but the key was not visible in the sandbox `env` field initially. Solution: wrote it to `/etc/environment` and `/root/.bashrc` via `dx()` calls, then sourced explicitly before `opencode run`.

3. **tmux not in snapshot**: `btf2go-validation:3` snapshot requires `apt-get install tmux` (was not pre-installed). After `apt-get update && apt-get install tmux`, worked fine.

4. **opencode output redirect**: `opencode run ... > /tmp/output.txt 2>&1` works but ANSI escape codes are still emitted. `sed 's/\x1b\[[0-9;]*m//g'` strips them cleanly for the transcript.

5. **Toolbox proxy URL**: The correct URL format is `https://proxy.app.daytona.io/toolbox/{sandbox-id}/process/execute` (not `https://proxy.app.daytona.io/toolbox/{sandbox-id}/toolbox/...`). The extra `/toolbox/` segment in the documented path causes 404.

### Ticket list (net-new from this run)

1. **[UX] `btf2go --version` fails**: `btf2go --version` returns "unknown flag: --version". Standard Go CLI convention is to support both `--version` and `-v`. A fresh user probing the installed version before use hits this. Fix: add a `version` subcommand or `--version` flag.

2. **[DOCS] Rust bitfield documentation gap**: The quickstart mentions "bitfields collapsed into `[N]byte` storage" but this refers to *C* bitfields. Rust has no native bitfield syntax. The quickstart should clarify that Rust users simulating bitfields need a plain `u32` field with manual accessors — btf2go will not generate `Get`/`Set` bitfield accessors from Rust structs (only from C `__attribute__((bitfield))`). M2.7 hallucinated `u5` trying to model this, which is a real user pain point.

3. **[UX] `btf2go generate` silent success**: When generate succeeds, there is no stdout output. A user expecting confirmation (even just a path echo) has to manually check. Adding `Generated: <path>` to stderr would reduce friction.

4. **[DOCS] `cp` directory pre-creation not obvious**: The quickstart shows copying events_gen.go but doesn't show `mkdir -p events/` first. Add that to step 5.

5. **[HARNESS] Daytona CLI org-header requirement**: The Daytona CLI v0.173.0 appears to not send the organization ID header when the profile's `api.key` is null and only `api.token` is set. This caused both CLI and MCP to fail with 401. Filed for awareness — may be a Daytona SDK bug.
